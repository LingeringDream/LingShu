# SoulLedger 设计决策增补(记忆 / 人格 / Thought Queue / 遗忘）

> 本文档是对 `AI-PersonalAssistant-PRD.md` 的增补,记录四个核心机制的设计取舍。
> 状态:已评审达成一致,部分已实现,剩余待排期。日期:2026-06。

## 0. 统一原则(贯穿全文)

**不要把 LLM 当成热路径里的打分/判断神谕。** 把"判断"赶到两个地方去:

1. **用户的显式信号**(滑块、点赞/点踩、"太啰嗦了"这类一键反馈)——信噪比最高。
2. **便宜的确定性启发式**(去重命中、计数、关键词、时间衰减)——可计算、可测试、零延迟。

只有在**能批处理、有完整上下文**的离线场景(如每晚一次的记忆整理)才请 LLM 做判断——
那时"LLM 没有上下文凭什么评估"的反驳不再成立。

推论:在没有真实用户数据前,所有权重/阈值/半衰期都是拍脑袋。**当前最该做的不是把公式调精,
而是把埋点做扎实**,让这些常数日后能从数据里长出来。每个机制下都列了"必须采集的信号"。

## 1. 记忆留存(retention)

**决策:Phase 0/1 不实现加权 `retention_score` 公式,改用极小决策树 + 现有机制。**

理由:仓库里已有两个机制天然逼近留存策略——
- `llm/dedup.rs` 近似去重:重复出现即 `GREATEST(importance)` 抬权(= repetition 信号的一半,已实现)。
- `llm/forgetting.rs` 时间衰减:不被复习的记忆自然沉底(已实现纯逻辑)。
两者合起来就是"用进废退",不需要先验地给七个维度配权重。

留存决策树(确定性,默认路径):
1. 用户显式"记住" → 高重要度保留(并置 `protect`)。
2. dedup 命中 → 抬权(现成)。
3. 否则默认 importance=0.5,交给遗忘衰减裁决。

LLM-as-judge **不删除,挪位置**:作为可选的离线"记忆整理"批处理(每晚一次),
在能看到该用户全部记忆的上下文下做精细化评分/合并。不进每轮热路径。

必须采集的信号(为日后校准):显式"记住"次数、dedup 命中次数、每条记忆的检索命中/复制/被引用次数。

## 2. 人格(personality)

**决策:重排可靠性优先级,降级"从行为自动演化数字"。**

根本问题:(a) 数字参数对不同 LLM 没有统一语义;(b) 行为信号有歧义
(复制 ≠ 认可风格,追问 ≠ 感兴趣)。凡是**信号源是推断来的行为**,都不可靠。

优先级(从可靠到不可靠):
1. **用户自己拉的滑块**(记忆/人格中心已支持编辑)→ 主路径,当主信号。
2. **显式反馈样本**:点赞/点踩、"太啰嗦/太正式"一键 chip。可存为 few-shot 正/负样本或拿来微调数字。
   *前置依赖:UI 需新增这些反馈入口(当前没有)。*
3. **从行为自动演化数字**(现 `llm/personality.rs`)→ 最危险,降级到最后,保留 24h 冷却 + 保守 clamp。

关于 few-shot 例子对(你的提案,方向对但非银弹):
- 代价:每轮注入 3-5 段完整问答很吃 token;正样本编码的是*内容*不只是*风格*,易让模型顺着老内容跑偏。
- 用法:只存"带风格标注的精简片段",不存整轮原文;每轮注入 3 条以内。

必须采集的信号:每条回复的赞/踩、"太长/太短/太正式"标签、用户手动改滑块的前后值。

## 3. Thought Queue(L7 主动建议)

**决策:不建常驻 worker。形态 = 事件触发 + 每日批处理 + 状态机。**

PRD 真正缺的不是"推理循环",是 **thought 的状态机**。

- **触发**:复用已有 post-stream 钩子(`thoughts.rs` 的 `should_generate_thoughts` 冷却)做即时触发,
  外加每日定时扫描兜底。两者都不是 always-on 循环。
- **"多次"的可计算定义**:跨 session 累计——某实体在 ≥3 个*不同对话*里出现,
  且无对应日历事件/任务 → 候选 thought。(借用 §1 的计数信号。)
- **防打扰(本质是通知疲劳问题)**:
  - thought 生命周期:`proposed → shown → accepted / dismissed / snoozed`(`thought_queue` 表应有 status,先确认)。
  - **触发器必须读状态**:被 dismiss 过的建议,用语义去重压制,X 天内不得重生。
  - 活跃 thought 数量上限;未处理的自动衰减。

必须采集的信号:thought 的展示/接受/忽略/snooze 事件、同一建议的重复生成次数。

## 4. 遗忘 vs 来源追溯(provenance)

**决策:遗忘 = 降级出工作集,不是销毁。provenance 优先于存储成本。**

先拆穿一个被混淆的概念:"遗忘"是"不再被检索",不是"被抹掉"。
`deleted_at` 软删除天然支持——被遗忘的记忆仍存在,只是排除出检索。冲突由此化解大半。

策略:
- **分层不删源**:N 条 episodic 压成 1 条 semantic 时,原始 episodic 保留(软降级),
  用 `source_memory_ids` 反向链接。用户可展开看"这条偏好从 5 个时刻推断而来",可逐条查看/删除。
- **遗忘扫描加护栏**:被任何*活跃* semantic 记忆的 `source_memory_ids` 引用的源,
  豁免遗忘(类似 `is_protected`,但基于引用关系)。否则会删掉支撑结论的证据,造成无法追溯的"幽灵偏好"。
- **成本取舍点在嵌入不在文本**:文本存储极便宜,贵的是向量索引。
  "遗忘"定义为:从 Qdrant 移除 + 排除检索,但**保留 PG 文本与 provenance**。成本与可追溯同时满足。

必须采集的信号:每条 semantic 记忆的 source 链、用户对推断偏好的"质疑/纠正"事件。

## 5. 实现顺序建议

1. 先补**埋点**(上述各节"必须采集的信号"),其余机制都依赖它来日后校准。
2. 记忆留存:落地决策树,接好 dedup+衰减(衰减 DB 接线已有提示词)。
3. 遗忘护栏:给遗忘扫描加 `source_memory_ids` 引用豁免。
4. 人格:加显式反馈 UI 入口 → 反馈样本 → (最后)再谈自动演化。
5. Thought Queue:补状态机 + 触发器读状态 + 每日批处理。
6. LLM-as-judge / 语义合并:作为离线批处理,最后做。
