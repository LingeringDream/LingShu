### R2. 20 周开发计划严重乐观

**问题**：5 个 Phase、20 周完成全部功能，几乎没有缓冲时间。

逐 Phase 分析：

| Phase | 计划时间 | 实际预估 | 问题 |
|-------|---------|---------|------|
| Phase 0 基础设施 | 2 周 | 2-3 周 | 可接受，但 Rust 项目的 CI/CD 比 Python 复杂 |
| Phase 1 核心对话 | 3 周 | 4-5 周 | Rust 生态无 LangChain 等价物，LLM 编排需自研 |
| Phase 2 虚拟形象 | 3 周 | **6-8 周** | 口型同步+表情系统+自定义系统，3 周不可能 |
| Phase 3 PM 智能 | 5 周 | **8-10 周** | 5 个子系统（集成/健康度/会议/图谱/报表），每个至少 2 周 |
| Phase 4 主动智能 | 4 周 | 5-6 周 | 风险预测模型需要数据积累和模型训练 |
| Phase 5 打磨 | 3 周 | 4-5 周 | 安全审计 + 用户测试至少需要 4 周 |
| **合计** | **20 周** | **29-37 周** | **低估 45-85%** |

**核心低估点**：
1. **3D 虚拟形象**：口型同步（音素映射 → BlendShape）是一个完整的子项目，不是 3 周能做完的功能
2. **Rust 生态缺乏 AI 框架**：没有 LangChain/LlamaIndex 的 Rust 等价物，对话管理、Prompt 模板、工具调用全部需要自研
3. **第三方集成**：每个平台（Jira/飞书/Asana/Slack/Teams...）的 API 适配至少需要 2-3 天
4. **知识图谱自动构建**：LLM 实体提取的准确率通常在 70-85%，需要大量后处理和人工校验逻辑

**建议**：
- 将 20 周调整为 **32-36 周**
- 或缩减 v1.0 范围：砍掉知识图谱、风险预测、报表自动生成，放到 v2.0
- 每个 Phase 预留 20% 缓冲时间

### R3. 资源预算遗漏 AI 服务的内存开销

**问题**：单用户部署预算写 "合计 ~510MB"，但这完全没算 AI 服务的内存。

实际内存需求：

| 组件 | 文档预算 | 实际需求 | 差距 |
|------|---------|---------|------|
| Ollama (Qwen2.5-7B Q4) | 未计入 | **4-6GB** | 巨大 |
| faster-whisper | 未计入 | **1-2GB** | — |
| TEI (向量模型) | 未计入 | **500MB-1GB** | — |
| **AI 服务合计** | **0** | **5.5-9GB** | — |
| **全栈真实合计** | **510MB** | **6-9.5GB** | **12-18x** |

如果用户选择纯云端 LLM（不部署 Ollama），内存可以降到 ~1.5GB，但文档没有说明这个关键前提。

**建议**：
- 资源预算表分为两种场景：**纯云端模式**（不部署本地模型）和 **本地推理模式**
- 纯云端模式下 ~1.5GB 全栈内存是可行的
- 本地推理模式下需要 8GB+ 内存，需明确告知用户

## 🟡 中等问题 (应该修复)

### R4. Apache AGE 的可行性风险被低估

**问题**：文档用 Apache AGE 替代 Neo4j 以节省内存，但未充分评估 AGE 的局限性。

AGE 的实际问题：
- **Cypher 语法不完整**：不支持 EXISTS 子查询、部分聚合函数、UNION 等
- **性能差距**：复杂多跳查询（3+ 跳）性能远不如 Neo4j
- **社区和文档**：远不如 Neo4j 成熟，遇到问题难以排查
- **运维复杂度**：需要在 PostgreSQL 上额外安装扩展，升级 PostgreSQL 版本时可能不兼容

**建议**：
- 将 AGE 标记为"实验性方案"，同时保留 Neo4j 作为备选
- 在 Phase 0 中增加一个 **技术验证 (PoC)** 环节：用 AGE 实现 3-5 个核心图查询，验证性能和功能是否满足需求
- 如果 AGE 不满足，回退到 Neo4j Community Edition（免费，内存 ~256MB 可控）

---

### R5. Rust 生态的 AI 集成成本被忽视

**问题**：文档假设 Rust 后端可以轻松集成各种 AI 服务，但 Rust 生态在 AI 领域有明显短板。

具体问题：
- **无成熟 LLM 编排框架**：Python 有 LangChain/LlamaIndex，Rust 没有等价物。对话管理、Prompt 模板、工具调用、链式推理全部需要手写
- **NLU 意图分类**：文档写"自研 NLU 模块"，但 Rust 的 NLP 生态非常薄弱，连基本的分词、词性标注都需要 FFI 调用 C 库
- **JSON Schema 验证**：LLM 返回的结构化数据需要严格验证，Rust 的 JSON Schema 库不如 Python 的 Pydantic 成熟

**建议**：
- 对话管理和 LLM 编排考虑用 Python 微服务实现，通过 gRPC 与 Rust 后端通信
- 或者接受 Rust 自研的成本，增加 4-6 周的开发时间用于构建基础 LLM 工具库
- 意图分类可以用简单的规则引擎 + 关键词匹配先跑起来，后续再迭代为 ML 模型

---

### R6. 安全章节过于简略

**问题**：安全与隐私章节只有 8 行文字，缺少具体实现方案。

遗漏的关键安全问题：
- **LLM Prompt 注入**：用户可以通过精心构造的输入让 LLM 执行非预期操作，文档完全没提及
- **API 密钥管理**：多个 LLM 提供商的 API Key 如何存储、轮换、泄露后如何应急？
- **第三方集成的 OAuth Token 管理**：Jira/飞书/Slack 的 Token 有过期、刷新、撤销机制
- **输入校验**：用户上传的文件可能包含恶意内容，如何在传给 LLM 前做安全检查？
- **多租户数据隔离**：团队部署时，项目 A 的管理员能否访问项目 B 的数据？

**建议**：新增一个完整的安全设计章节，覆盖上述所有问题。

---

### R7. 缺少错误处理和降级策略

**问题**：文档有自适应降级策略（A.7.2），但只覆盖了"系统资源紧张"的场景。缺少以下关键场景的处理：

- LLM API 超时或返回错误 → 用户体验如何？
- 第三方平台 API 限流 → 如何排队和重试？
- 向量数据库不可用 → 记忆检索降级为什么方案？
- WebSocket 断连 → 如何恢复会话状态？
- 数据库主从切换 → 如何保证不丢数据？

**建议**：新增错误处理矩阵，每种故障场景都有明确的降级路径和用户提示。

---

### R8. 前后端技术栈割裂

**问题**：前端用 TypeScript/React，后端用 Rust。两个语言之间没有共享的类型定义。

这意味着：
- API 请求/响应的类型需要在两端分别维护，容易不一致
- WebSocket 消息格式也需要两端同步
- 团队需要同时精通 TypeScript 和 Rust，招聘难度大

**建议**：
- 使用 OpenAPI (utoipa 生成) + openapi-typescript-codegen 自动生成前端 API 客户端
- 或使用 Protocol Buffers 作为跨语言契约
- 在 CI 中加入 API 契约测试，确保前后端类型一致

## 🟢 建议改进 (锦上添花)

### R9. 缺少 API 规范

文档没有定义任何具体的 API 端点。建议至少列出核心 API：

```
POST /api/chat              — 发送消息（支持流式响应）
GET  /api/projects           — 获取项目列表
GET  /api/projects/:id       — 获取项目详情
GET  /api/projects/:id/health — 获取项目健康度
POST /api/meetings/transcribe — 上传会议录音转录
GET  /api/tasks              — 获取任务列表
POST /api/reports/generate   — 生成报告
WS   /ws/chat                — WebSocket 实时对话
```

### R10. 缺少测试策略

文档的 Checklist 只有 "cargo test 全通过"，但没有说明：
- 单元测试覆盖率目标（建议 ≥ 80%）
- 集成测试策略（如何 mock LLM API？）
- E2E 测试（3D 渲染如何自动化测试？）
- 性能基准测试（如何持续监控性能回退？）

### R11. 缺少数据库迁移策略

PostgreSQL schema 会随版本迭代变化，文档没有说明：
- 使用什么迁移工具（建议 sqlx-cli 或 refinery）
- 迁移脚本的版本管理
- 向前兼容策略（零停机迁移）

### R12. 知识图谱自动构建的准确率问题

文档写"自动从对话/文档中提取实体和关系"，但：
- LLM 实体提取的准确率通常 70-85%，会产生大量错误关系
- 没有人工审核/确认机制
- 错误的图谱数据会被长期记忆，影响后续推理

**建议**：增加"图谱写入前需用户确认"的机制，或至少标记为"待确认"状态。

### R13. 商业模式需要更务实

"社区版免费 + 专业版 ¥99/月"的定价需要考虑：
- LLM API 调用成本（GPT-4o 约 ¥0.07/1K tokens，重度用户月成本可能超过 ¥99）
- 需要计算盈亏平衡点
- 建议先做成本模型，再定价格

## 审查结论

这份 PRD 的**产品定位和功能设计是出色的**——PM 专属 AI 助理 + 3D 虚拟形象 + 主动智能的组合确实有差异化。技术架构的**方向也是正确的**——Rust 后端的性能优势真实存在。

但文档在**工程落地层面**存在明显的乐观偏差：时间规划低估了 45-85%，资源预算遗漏了最大的内存消耗项（AI 服务），技术栈切换后多处代码示例未同步更新。

建议在启动开发前：
1. 修复上述 🔴 严重问题
2. 做一次 Phase 0 的技术验证 (PoC)，特别是 Apache AGE 和 Rust LLM 编排的可行性
3. 根据 PoC 结果重新评估时间线和资源需求
