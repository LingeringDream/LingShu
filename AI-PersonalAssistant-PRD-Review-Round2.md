## 🟡 中等问题 (本轮新发现)

### R15. 商业模式定价不一致

**问题**：专业版定价在两处给出不同数字。

- 第 724 行（商业模式表格）：`专业版 ¥99/月`
- 第 742 行（成本模型）：`建议调至 ¥149-199/月`
- 第 744 行（盈亏平衡）：`专业版定价 ¥199/月`

读者不知道以哪个为准。

**建议**：统一为一个数字。根据成本模型分析，建议将商业模式表格中的 ¥99/月改为 ¥199/月，并在旁边标注"详见 9.3 成本模型"。

### R16. Phase 3 范围仍然过载

**问题**：Phase 3（第 15-22 周，8 周）包含 5 个子系统，工作量仍然偏大。

逐项分析：
- Jira + 飞书集成：每个平台的 OAuth + API 适配 + Webhook 处理约 2 周 → 共 4 周
- 项目健康度引擎：指标定义 + 计算 + 可视化约 2 周
- 会议助手：录音转录 + 纪要生成 + Action Item 提取约 2 周
- 通知引擎 + 调度引擎：约 2 周

合计约 10 周，但只分配了 8 周。

**建议**：
- Phase 3a（第 15-19 周）：Jira 集成 + 健康度引擎 + 通知引擎
- Phase 3b（第 20-24 周）：飞书集成 + 会议助手 + 调度引擎
- 总工期从 34 周调整为 **36 周**

---

### R17. 前端 3D 渲染的 E2E 测试策略缺失

**问题**：测试策略章节覆盖了后端测试（cargo test、wiremock、criterion.rs），但 3D 虚拟形象的自动化测试完全没有提及。

3D 渲染的测试难点：
- Three.js 渲染结果无法用传统断言验证
- 口型同步的正确性如何自动化验证？
- 不同 GPU/浏览器的渲染一致性如何保证？

**建议**：
- 截图对比测试：Playwright 截图 + pixelmatch 像素对比
- 口型同步：录制音频 → 对比 BlendShape 关键帧序列的数值断言
- 性能测试：WebGL 帧率监控 + 内存增长检测
- 浏览器矩阵：Chrome/Firefox/Safari 核心渲染测试

### R19. 缺少数据备份与灾难恢复方案

**问题**：文档覆盖了数据安全（加密、权限），但没有提及：
- 数据库备份策略（全量/增量/频率）
- 灾难恢复流程（RTO/RPO 目标）
- 向量数据库和图数据库的备份策略

**建议**：新增简短的备份策略章节：
- PostgreSQL：每日全量备份 + WAL 归档实现 PITR（Point-in-Time Recovery）
- Qdrant：快照备份（Qdrant 支持 collection snapshot）
- Redis：RDB + AOF 持久化
- RTO 目标：< 1 小时，RPO 目标：< 5 分钟

## 🟢 建议改进

### R21. API 版本化策略未提及

当前 API 路径是 `/api/chat`、`/api/projects`，没有版本前缀。未来 API 变更时如何保持向后兼容？

**建议**：使用 `/api/v1/` 前缀，或在 Accept Header 中指定版本。

### R22. 错误响应格式未统一

API 规范列出了端点，但没有定义统一的错误响应格式。

**建议**：定义标准错误格式：
```json
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "请求过于频繁，请稍后再试",
    "details": { "retry_after": 30 }
  }
}
```

### R23. 前端 WebSocket 与 REST API 的职责边界

文档同时定义了 `POST /api/chat` (REST) 和 `WS /ws/chat` (WebSocket) 两种对话方式。何时用哪个？

**建议**：明确说明：
- REST：单次请求-响应、报表生成、管理操作
- WebSocket：实时对话、流式输出、虚拟形象状态同步
